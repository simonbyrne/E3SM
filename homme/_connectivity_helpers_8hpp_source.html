19 // +--------------------------------------------------------------------------------------------------------------+
20 // | |
21 // | REMARKS ON BOUNDARY EXCHANGE |
22 // | |
23 // | Each element has 8 neighbors: west(W), east(E), south(S), north(N), |
24 // | south-west(SW), south-east(SE), north-west(NW), north-east(NE) |
25 // | The first 4 correspond to the neighbors sharing a full edge with this element, |
26 // | while the latters correspond to the neighbors sharing only a corner with this element. |
27 // | NOTE: if the # of elements on each face of the cube sphere (ne*ne in homme) is greater than one, then |
28 // | there are 24 elements that miss one corner neighbor. These are the element that touch one |
29 // | of the cube vertices (if ne=1, then all elements miss all the corner neighbors). |
30 // | The numeration of dofs on each element is the following |
31 // | |
32 // | (NW) -(N)-> (NE) |
33 // | 12--13--14--15 |
34 // | | | | | |
35 // | | | | | |
36 // | ^ 8---9--10--11 ^ |
37 // | | | | | | | |
38 // | (W) | | | | (E) |
39 // | | 4---5---6---7 | |
40 // | | | | | |
41 // | | | | | |
42 // | 0---1---2---3 |
43 // | (SW) -(S)-> (SE) |
44 // | |
45 // | The arrows around an edge neighbor refer to the 'natural' ordering of the dofs within an edge. |
46 // | From the picture we can immediately state the following: |
47 // | |
48 // | 1) edge neighbors contain 4 points/dofs, while corners contain only 1 |
49 // | 2) the S/N edges store the points contiguously, while the W/E points are strided (stride is NP) |
50 // | (Note: this is relevant only if we decide to switch to RMA, avoiding buffers, and copying |
51 // | data directly to/from host views) |
52 // | |
53 // | In addition, we need to understand that the local ordering of edge points may differ on two |
54 // | neighboring elements. For instance, consider two elements sharing their S edge. This is depicted |
55 // | in the following: |
56 // | |
57 // | elem 1 |
58 // | 0---1---2---3 |
59 // | |
60 // | 3---2---1---0 |
61 // | elem 2 |
62 // | |
63 // | So the 1st dof on the S edge of elem 1 does not correspond to the 1st dof on the S edge of elem 2. |
64 // | This always happen for same-edge neighbors (W/W, E/E, S/S/, N/N), and also for W/N, E/S. For these |
65 // | neighbors we need to store a flag marking the neighbor as 'backward' ordered. The other neighbors |
66 // | (S/W, S/N, W/E, E/N) are marked as 'forward', meaning that the 1st dof on elem1's edge matches the |
67 // | 1st dof on elem2's edge. |
68 // | NOTE: in F90, in this context, 'edge' means an edge of the dual mesh, i.e., a connection between elements. |
69 // | Here, we reserve the word 'edge' for the mesh edges, and we use 'connection' to refer to the 8 |
70 // | possible connections with neighboring elements. |
71 // | |
72 // | The W/E/S/N/SW/SE/NW/NE values from a neighboring element are stored on a buffer, to allow all MPI |
73 // | operations to be over before the local view is updated. The values are then summed into the local field |
74 // | view in a pre-established order, to guarantee reproducibility of the accumulation. |
75 // | |
76 // +--------------------------------------------------------------------------------------------------------------+
